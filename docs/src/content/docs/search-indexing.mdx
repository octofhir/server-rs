This page explains the practical indexing model used by OctoFHIR today.

If you only need query syntax, see [Search Parameters](./search-parameters).  
If you need internals (registry/cache lifecycle), see [Search Architecture](./search-architecture).

## Quick Summary

OctoFHIR uses native search IR backed by denormalized index tables for
production search paths:

1. **Denormalized index tables** for high-impact search types:
   - `search_idx_reference`
   - `search_idx_date`
   - `search_idx_string`
   - `search_idx_number`
   - `search_idx_quantity`
2. **Direct JSONB SQL paths** only for intentionally non-indexed surfaces such
   as narrative full-text (`:text`).

This keeps FHIR search semantics explicit while avoiding runtime JSONB fallback
for production query types.

## What Is Indexed

| Query Type | Primary Execution Path | Notes |
|---|---|---|
| Reference search (`subject=Patient/123`) | `search_idx_reference` | Optimized for lookup and joins |
| `_include` / `_revinclude` | `search_idx_reference` | Uses normalized reference rows |
| Chained search | `search_idx_reference` + target query | Efficient join start point |
| Date search (`ge`, `lt`, `ap`, etc.) | `search_idx_date` | Precision-aware ranges |
| String default / `:contains` / `:exact` | `search_idx_string` | Normalized and exact values |
| Number search | `search_idx_number` | Prefix-aware numeric ranges |
| Quantity search | `search_idx_quantity` | Numeric value plus system/code/unit |
| Token simple code (`gender=female`) | JSONB containment (`@>`) | Uses the resource GIN index |

## Debug Plans

Search debug output is disabled by default and must be enabled explicitly:

```toml
[search]
allow_debug_search_plan = true
allow_debug_search_explain_analyze = false
```

When enabled, request a redacted search plan with either query syntax or the
header form:

```http
GET /fhir/Patient?family=Smith
X-OctoFHIR-Search-Debug: plan
```

Accepted header values are `plan`, `sql`, `explain`, and `explain-analyze`.
`explain-analyze` additionally requires `allow_debug_search_explain_analyze =
true` because it executes the query. The response is returned as a FHIR
`Parameters` resource and redacts search values while preserving strategy,
index, and SQL-shape information.

## How This Affects Query Design

### Prefer Typed References

```bash
# Better (typed)
GET /Observation?subject=Patient/123

# Allowed but less explicit
GET /Observation?subject=123
```

### Use Bounded Date Windows

```bash
GET /Observation?date=ge2026-01-01&date=lt2026-02-01
```

Bounded windows usually plan better than very broad unbounded date filters.

### Use `:exact` Intentionally

```bash
GET /Patient?family:exact=Smith
```

Use `:exact` for exact text semantics, not as a generic performance toggle.

## Write Path Behavior

On resource create/update:

1. Search parameters for the resource type are resolved.
2. Indexed values are extracted from resource JSON.
3. Existing rows for that resource are replaced in the relevant `search_idx_*` tables.

On delete:

- Rows are removed from all search index tables for that resource.

## Operational Checks

### Verify index tables

```sql
\dt search_idx*
```

Expected:

- `search_idx_reference`
- `search_idx_date`
- `search_idx_string`
- `search_idx_number`
- `search_idx_quantity`

### Check cardinality quickly

```sql
SELECT 'reference' AS kind, count(*) FROM search_idx_reference
UNION ALL
SELECT 'date' AS kind, count(*) FROM search_idx_date
UNION ALL
SELECT 'string' AS kind, count(*) FROM search_idx_string
UNION ALL
SELECT 'number' AS kind, count(*) FROM search_idx_number
UNION ALL
SELECT 'quantity' AS kind, count(*) FROM search_idx_quantity;
```

## Tuning Strategy

Start with defaults first. Add custom indexes only when measurements justify them.

Typical triggers for custom indexes:

- very high traffic on one string field
- workload-specific extension filters
- clear slow-query evidence + repeatable EXPLAIN improvement

For broader guidance, see [Performance Tuning](./performance).
